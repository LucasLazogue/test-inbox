

## Summary

Updates the entire color palette of the `test-inbox` application (a React + Vite LangGraph POC) to Peñarol brand colors — primary yellow (`#FFCC00`) and secondary black (`#000000`). The change is concentrated in `style.css` (CSS custom properties and all hardcoded color values) with minor inline-style adjustments in `src/App.jsx`. The result is a fully rebranded UI where backgrounds, text, buttons, chat bubbles, borders, scrollbars, and status indicators all reflect the Peñarol yellow-and-black scheme.

## Problem Statement

In `style.css` (lines 1–39), the application uses a dark-navy/green-accent palette defined via CSS custom properties and numerous hardcoded color literals:

```css
:root{
  --bg:#0f1724;          /* dark navy */
  --card:#0b1220;
  --accent:#6ee7b7;      /* green accent */
  --muted:#9aa4b2;
  --glass: rgba(255,255,255,0.03);
  --bubble-in:#0b6e4f;   /* green bubble */
  --bubble-out:#1f2937;
}
body{background:linear-gradient(180deg,#071023 0%, #081226 100%);color:#e6eef6}
```

Additionally, `src/App.jsx` (line 180) contains an inline `style` object on the composer `<div>`, and the `Message` component uses CSS classes (`pill`, `widget-boolean`) whose colors are hardcoded in `style.css` (lines 28, 32) with green tones.

**Concrete consequence:** The UI currently presents a navy-blue/green brand identity that has no relation to Peñarol. Every color token, gradient, bubble color, button background, and text tint must be replaced to achieve brand alignment with Peñarol's yellow (`#FFCC00`) and black (`#000000`).

## Goals / Non-Goals

### Goals
1. All CSS custom properties in `:root` use Peñarol-derived yellow/black values.
2. Every hardcoded color literal in `style.css` is replaced with a Peñarol-palette equivalent.
3. The `.primary` button class renders with yellow (`#FFCC00`) background and black text.
4. Chat bubbles (`.msg.in`, `.msg.out`) use yellow/black tones instead of green/navy.
5. Boolean widget pills (`.widget-boolean .pill`) use yellow/black instead of green.
6. Body background is black/near-black; primary text is light (white or light-yellow for contrast).
7. No references to the old palette colors (`#6ee7b7`, `#0b6e4f`, `#0f1724`, `#071023`, `#081226`, `#0b1220`, `#134b3a`, `#1f2937`, `#042016`, `#0f3e29`, `#083421`, `#dff7ea`) remain in the codebase.
8. The built `dist/` output reflects the new palette after running `npm run build`.

### Non-Goals
- Changing the application layout, typography (font-family), or component structure.
- Adding a theme-switcher or dark/light mode toggle.
- Modifying any API endpoints, business logic, or JavaScript behavior in `App.jsx`.
- Updating the `dist/` folder manually (it will be regenerated by `vite build`).
- Adding a CSS preprocessor (Sass, PostCSS, etc.) — the project uses plain CSS.

## Proposed Solution

### Architecture Overview

The application has a minimal architecture:

```
index.html          ← Vite entry point (references #root)
src/main.jsx        ← React bootstrap, imports /style.css
src/App.jsx         ← Single-file React app (App + Message components)
style.css           ← ALL styling (CSS custom properties + rules)
vite.config.js      ← Vite config (proxy rules)
dist/               ← Build output (regenerated, not manually edited)
```

All color definitions live in **one file**: `style.css`. The React components in `src/App.jsx` reference CSS classes defined there. There is one inline style in `App.jsx` (line 180) that uses no color values — only layout (`marginTop`, `display`, `gap`, `alignItems`), so it requires no change.

The change is a **pure CSS token replacement** with no structural modifications.

## Risks

1. **Contrast / Accessibility:** Yellow (`#FFCC00`) on white or light backgrounds has poor contrast (WCAG AA fail). Mitigated by using black backgrounds throughout and black text on yellow surfaces. The `.msg.in` bubble (yellow background) explicitly sets `color:#000000`. The `.meta` text inside incoming bubbles inherits black — verify it remains readable.

2. **Muted text readability:** The new `--muted` value (`#b3a44e`, muted gold) on near-black backgrounds should be checked for WCAG AA compliance. Contrast ratio of `#b3a44e` on `#0a0a0a` is approximately 6.2:1 — passes AA for normal text. If it feels too dim, consider `#c4b55f` (lighter gold).

3. **`.pill` default color vs `.widget-boolean .pill`:** The generic `.pill` now has `color:#FFCC00` (yellow text on semi-transparent yellow background). The `.widget-boolean .pill` overrides this with solid yellow background + black text. Verify that non-boolean pills (e.g., "Mark Done", "Enviar" inside widget-text) are readable — yellow text on `rgba(255,204,0,0.15)` over a dark parent should be fine.

4. **Hardcoded inline styles in `App.jsx`:** Line 180 has inline styles but no color values. However, the `<input className="short">` on line 181 and `<button className="primary">` on line 189 rely on CSS classes — confirmed they will pick up the new palette automatically.

5. **Browser-specific scrollbar styling:** The `::-webkit-scrollbar-thumb` rule only affects WebKit/Blink browsers. Firefox users will see the default scrollbar. This is pre-existing behavior, not a regression.

6. **`dist/` stale assets:** If the developer forgets to run `npm run build` after the CSS change, the `dist/` folder will still contain old-palette assets. The migration plan explicitly includes this step.

## Open Questions

1. **Exact yellow shade:** The spec uses `#FFCC00` as the primary Peñarol yellow. The official Peñarol yellow varies by source (some use `#FFD100`, others `#FDD835`). Should we match a specific official brand guide, or is `#FFCC00` confirmed as the desired value?

2. **Hover / focus states:** The current CSS has no `:hover` or `:focus` styles for buttons or pills. Should this spec include adding hover states (e.g., darker yellow `#e6b800` on hover for `.primary`)? This would be a minor enhancement beyond the palette swap.

3. **Favicon / meta theme-color:** The `index.html` may have a `<meta name="theme-color">` or favicon that should also reflect the new brand. Should this be included in scope?
